# Vectors

pgvector brings first-class vector embedding storage and similarity search to your MimDB project. Store, index, and query high-dimensional embeddings without a separate vector database.

## What is pgvector

pgvector is a PostgreSQL extension that adds a native `vector` column type and a set of distance operators for similarity search. Every vector table you create through MimDB is a real PostgreSQL table in your project's schema, with Row Level Security enabled automatically.

Use cases include:

- Semantic search over documents or products
- Recommendation systems driven by embedding similarity
- Image or audio search with model-generated embeddings
- Retrieval-Augmented Generation (RAG) pipelines

## Enabling pgvector

pgvector must be installed at the PostgreSQL system level before it can be enabled in MimDB. Once installed, turn it on in `config.toml`:

```toml
[extensions.pgvector]
enabled = true
default_dimensions = 1536        # used when dimensions are omitted at table creation
default_distance_metric = "cosine"  # cosine | l2 | inner_product
```

<Note>
pgvector requires a system-level installation on the PostgreSQL server (`CREATE EXTENSION vector` must succeed). If the extension is not installed, all `/v1/vectors/` endpoints return `VEC-0001: pgvector not enabled on this deployment`.
</Note>

After enabling, MimDB exposes the full vectors API under `/v1/vectors/{ref}/` for your project.

## HNSW indexing

MimDB creates an HNSW (Hierarchical Navigable Small World) index on the `embedding` column automatically when you create a vector table. HNSW is an approximate nearest-neighbor algorithm that trades a small amount of recall for dramatically faster search at scale.

Two parameters control the index:

| Parameter | Default | Range | Effect |
|-----------|---------|-------|--------|
| `m` | `16` | 2-100 | Max connections per layer. Higher = better recall, more memory and build time. |
| `ef_construction` | `64` | 1-500 | Search depth during index build. Higher = better recall, slower build. |

To use custom parameters, pass an `index` object when creating the table:

```json
{
  "name": "documents",
  "dimensions": 1536,
  "metric": "cosine",
  "index": {
    "m": 32,
    "ef_construction": 128
  }
}
```

To defer index creation entirely (useful for bulk-loading data before indexing), set `skip_index: true` in the create request and call `POST /v1/vectors/{ref}/{table}/index` after loading.

<Tip>
The defaults (`m=16`, `ef_construction=64`) are appropriate for most workloads up to a few million rows. For high-recall requirements or very large tables, increase `ef_construction` to 128-256 and `m` to 32-64. Always benchmark with your actual data before tuning in production.
</Tip>
