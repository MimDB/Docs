# RPC Functions

Any PostgreSQL function created in your project's schema is automatically
callable over HTTP via the RPC endpoint. This lets you run complex server-side
logic, aggregations, or multi-step operations in a single request.

## What are RPC functions

PostgREST exposes PostgreSQL functions at:

```
POST /v1/rest/{ref}/rpc/{function_name}
```

The function must exist in the project's schema (the schema MimDB provisions for
your project). Its parameters map directly to the JSON body of the request. The
return value is serialised as the HTTP response body.

<Note>
RPC functions run inside PostgREST's connection, which uses the caller's
PostgreSQL role. Row Level Security policies and `current_setting` JWT claims are
active for the duration of the function call, exactly as they are for regular
table reads and writes.
</Note>

## Creating a function

Use the SQL execution endpoint to create a function in your project's schema.
This requires a `service_role` API key or a `service_role` JWT.

<Steps>

<div>

### Write the function

Write a PostgreSQL function using standard SQL. This example counts incomplete
todos and returns the result as an integer:

```sql
CREATE OR REPLACE FUNCTION count_pending_todos()
RETURNS integer
LANGUAGE sql
STABLE
AS $$
  SELECT count(*)::integer FROM todos WHERE done = false;
$$;
```

</div>

<div>

### Create it via the SQL endpoint

Send the `CREATE FUNCTION` statement to `POST /v1/sql/{ref}/execute`:

<CodeGroup>
```bash cURL
curl -X POST "http://localhost:9000/v1/sql/a1b2c3d4/execute" \
  -H "Content-Type: application/json" \
  -H "ApiKey: <service_role_key>" \
  -d '{
    "query": "CREATE OR REPLACE FUNCTION count_pending_todos() RETURNS integer LANGUAGE sql STABLE AS $$ SELECT count(*)::integer FROM todos WHERE done = false; $$;"
  }'
```
</CodeGroup>

A successful response returns `200 OK` with an empty result set.

</div>

<div>

### Call the function

Once created, the function is immediately available at the RPC endpoint:

<CodeGroup>
```bash cURL
curl -X POST "http://localhost:9000/v1/rest/a1b2c3d4/rpc/count_pending_todos" \
  -H "Content-Type: application/json" \
  -H "ApiKey: <anon_key>" \
  -d '{}'
```
</CodeGroup>

**Response**

```json
3
```

</div>

</Steps>

## Calling a function

Send a `POST` request to `/v1/rest/{ref}/rpc/{function_name}` with a JSON object
whose keys match the function's parameter names.

<CodeGroup>
```bash No parameters
curl -X POST "http://localhost:9000/v1/rest/a1b2c3d4/rpc/count_pending_todos" \
  -H "Content-Type: application/json" \
  -H "ApiKey: <anon_key>" \
  -d '{}'
```

```bash With parameters
# Function: add_todo(task text, priority int) RETURNS void
curl -X POST "http://localhost:9000/v1/rest/a1b2c3d4/rpc/add_todo" \
  -H "Content-Type: application/json" \
  -H "ApiKey: <anon_key>" \
  -d '{
    "task": "Review PR",
    "priority": 1
  }'
```

```bash Returning a table
# Function: active_todos() RETURNS SETOF todos
curl -X POST "http://localhost:9000/v1/rest/a1b2c3d4/rpc/active_todos" \
  -H "Content-Type: application/json" \
  -H "ApiKey: <anon_key>" \
  -d '{}'
```
</CodeGroup>

**Response for table-returning function**

```json
[
  { "id": 1, "task": "Read the MimDB docs", "done": false, "created_at": "2025-01-01T12:00:00Z" },
  { "id": 3, "task": "Write unit tests",    "done": false, "created_at": "2025-01-02T09:00:00Z" }
]
```

## Return types

PostgREST handles each PostgreSQL return type differently:

| Return type | HTTP response | Notes |
|-------------|--------------|-------|
| Scalar (`integer`, `text`, etc.) | JSON primitive | `3`, `"hello"` |
| `void` | `204 No Content` | No response body |
| `record` or composite type | JSON object | `{"col": value, ...}` |
| `SETOF <table>` | JSON array | Same format as `GET` on the table |
| `TABLE(...)` | JSON array | Inline return type definition |
| `json` / `jsonb` | Raw JSON value | Passed through as-is |

Functions returning `SETOF` or `TABLE(...)` support the full PostgREST query
interface: you can filter, order, and paginate the results using the same query
parameters as table reads:

<CodeGroup>
```bash Paginate and filter a set-returning function
curl "http://localhost:9000/v1/rest/a1b2c3d4/rpc/active_todos?done=eq.false&order=created_at.desc&limit=5" \
  -H "ApiKey: <anon_key>" \
  -d '{}'
```
</CodeGroup>

## Security

**Role execution.** Functions run under the PostgreSQL role resolved from the
caller's credential (the same role used for table access). RLS policies apply to
any tables the function queries.

**SECURITY INVOKER vs SECURITY DEFINER.**

| Attribute | Behaviour |
|-----------|-----------|
| `SECURITY INVOKER` (default) | Function runs as the caller's role. RLS policies apply normally. |
| `SECURITY DEFINER` | Function runs as the role that defined it (typically a privileged role). RLS is bypassed for that role's tables. |

<Note>
Use `SECURITY DEFINER` with care. Because the function runs as its owner's role,
it can bypass Row Level Security on any table that role can access. Prefer
`SECURITY INVOKER` for functions that should respect the caller's permissions.
When `SECURITY DEFINER` is necessary (for example, to insert into an audit log
that users cannot modify directly), explicitly set the `search_path`:

```sql
CREATE FUNCTION my_func() RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = ''
AS $$ ... $$;
```

Setting an empty `search_path` prevents search-path injection attacks.
</Note>
