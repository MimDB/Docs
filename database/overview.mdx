# Database

Every MimDB project gets its own dedicated PostgreSQL database. Tables you
create in that database are immediately available as REST endpoints through the
managed PostgREST sidecar - no extra configuration required.

## Multi-tenant architecture

MimDB uses **one PostgreSQL database per project**. When you create a project
with ref `ca1deb46032e8010`, the platform provisions a database with that name
and three PostgreSQL roles scoped to it:

| Role | Purpose |
|------|---------|
| `ca1deb46032e8010_anon` | Unauthenticated requests, subject to RLS |
| `ca1deb46032e8010_authenticated` | JWT-bearing requests, subject to RLS |
| `ca1deb46032e8010_service` | Service role, bypasses RLS |

All user tables live in the `public` schema of the project database. Every
data-plane request (`/v1/rest/`, `/v1/realtime/`, `/v1/storage/`) runs inside a
transaction where `SET LOCAL ROLE` is applied from the JWT claim. Because each
project has its own database, one project can never read or write rows in
another project's database.

RLS helper functions (`auth.uid()`, `auth.role()`) are installed in each project
database, and an auto-RLS event trigger ensures new tables inherit the correct
baseline policies.

<Note>
The platform's own management tables live in the `_mimisbrunnr` schema of the
host database and are never exposed through PostgREST or the SQL endpoint.
</Note>

## Schema introspection

MimDB exposes two read-only introspection endpoints that query PostgreSQL
system catalogs (`pg_class`, `pg_attribute`, `pg_constraint`, `pg_index`).
Both require a `service_role` key.

### List tables

```bash
curl http://localhost:9000/v1/introspect/<ref>/tables \
  -H "Authorization: Bearer <service_role_key>"
```

```json
{
  "data": [
    {
      "name": "todos",
      "schema": "public",
      "row_estimate": 42,
      "size_bytes": 8192,
      "comment": null
    }
  ],
  "error": null,
  "meta": { "request_id": "01970000-0000-7000-8000-000000000000" }
}
```

## How PostgREST exposes tables

The PostgREST sidecar inspects the project database on startup and after every
schema reload. Any object visible to the `_mimisbrunnr_authenticator` role
in the `public` schema becomes a REST endpoint under `/v1/rest/{ref}/`:

- **Tables** map to `/v1/rest/{ref}/{table_name}`
- **Views** are exposed the same way as tables
- **Stored functions** are callable via `/v1/rest/{ref}/rpc/{function_name}`

Row-level security policies are enforced automatically. The `anon` role sees
only rows permitted by your RLS policies; the `service_role` key bypasses RLS
entirely.

<Note>
Schema changes (new tables, added columns, dropped indexes) take effect after
PostgREST performs a schema reload. MimDB tracks schema versions and triggers
reloads automatically; most DDL changes are visible within seconds.
</Note>
