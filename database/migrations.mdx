# Migrations

MimDB separates two migration concerns: **platform migrations** that run at
server startup and keep the shared infrastructure schema up to date, and
**custom SQL** that you run against your project's tenant schema using the
SQL execution endpoint.

## Platform migrations

Platform migrations live in the server binary and are applied automatically on
every startup using a sequential versioning scheme. No manual intervention is
required - the server will not accept requests until all pending migrations have
been applied.

<Note>
Platform migrations are idempotent. Restarting the server after a failed
migration or an upgrade is always safe; already-applied migrations are
skipped.
</Note>

## Tenant template

When a new project is created, MimDB executes the tenant template SQL inside
the new project schema. This provisions the initial objects every tenant needs
before any application tables are created:

- `auth_users` - user accounts with email, phone, display name, and metadata
- `auth_refresh_tokens` - refresh tokens with family-based reuse detection
- `auth_identities` - OAuth provider identities linked to users
- `auth_password_resets` - password reset tokens
- `auth_sessions` - active session tracking with device metadata
- RLS enabled on all auth tables
- RLS policies restricting auth table access to the row owner
- All auth tables revoked from the `anon` and `authenticated` roles (access goes through the Go auth service, not PostgREST)

Your application tables are separate from these. The auth tables are managed by
MimDB's auth subsystem; you create your own tables independently using the SQL
execution endpoint.

## Custom SQL

The `POST /v1/sql/{ref}/execute` endpoint runs arbitrary SQL against the tenant
schema. It requires the `service_role_key`.

<CodeGroup>
```bash Single statement
curl -X POST http://localhost:9000/v1/sql/<ref>/execute \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <service_role_key>" \
  -d '{
    "query": "CREATE TABLE events (id uuid primary key default gen_random_uuid(), name text not null, payload jsonb, occurred_at timestamptz not null default now())"
  }'
```

```bash Parameterized query
curl -X POST http://localhost:9000/v1/sql/<ref>/execute \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <service_role_key>" \
  -d '{
    "query": "INSERT INTO events (name, payload) VALUES ($1, $2)",
    "params": ["user.created", {"user_id": "abc123"}]
  }'
```

```bash Transaction block
curl -X POST http://localhost:9000/v1/sql/<ref>/execute \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <service_role_key>" \
  -d '{
    "query": "BEGIN; ALTER TABLE events ADD COLUMN source text; ALTER TABLE events ADD COLUMN severity text NOT NULL DEFAULT '"'"'info'"'"'; COMMIT;"
  }'
```
</CodeGroup>

**Limits and behavior:**

| Property | Value |
|----------|-------|
| Maximum query size | 64 KB |
| Statement timeout | Configurable via `[pgdog].statement_timeout` (default 30 s) |
| Maximum result rows | 10,000 (result is truncated, not errored) |
| `search_path` | Set to the project schema for every execution |
| Role | `service_role` - RLS is not enforced |

The timeout is enforced via `SET LOCAL statement_timeout` inside a transaction,
so it reverts cleanly when the connection returns to the pool. If the timeout
fires, the endpoint returns `408` with error code `SQL-0005`.

Use parameterized queries (`$1`, `$2`, ...) when inserting user-supplied values
to prevent SQL injection. The `params` array maps positionally to the
placeholders in the query string.

## Best practices

<Warning>
Destructive migrations (`DROP TABLE`, `DROP COLUMN`, `TRUNCATE`) cannot be
rolled back once committed. Always take a database backup before running them
in production.
</Warning>

- **Version your migrations.** Keep migration SQL in numbered files in your
  application repository (e.g., `0001_create_events.sql`). Apply them in order
  and track which have run in a `schema_migrations` table.

- **Test on staging first.** Run every migration against a staging project
  before applying it to production. Schema errors are far cheaper to debug
  before they touch real data.

- **Use transactions for multi-statement changes.** Wrap related DDL in
  `BEGIN` ... `COMMIT` so that a failure midway leaves the schema unchanged
  rather than partially modified.

- **Avoid long-held locks.** Operations like `ALTER TABLE ... ADD COLUMN NOT NULL`
  without a default lock the table for the duration. On large tables, use
  `ADD COLUMN` with a default first, then backfill, then add the constraint
  separately.
