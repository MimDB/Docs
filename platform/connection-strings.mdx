# Connection Strings

MimDB can expose direct PostgreSQL access to your project's database through the
PgDog connection pooler. This is useful for running migrations, analytics
queries, ORMs, or any tooling that needs a real PostgreSQL connection rather
than the REST API.

## Connection info

`GET /v1/platform/projects/{id}/connection-info`

Returns the connection parameters for a project. The response includes the
pooled connection (via PgDog) when configured, and optionally a direct
PostgreSQL connection.

<CodeGroup>
```bash Request
curl http://localhost:9000/v1/platform/projects/01970000-0000-7000-8000-000000000002/connection-info \
  -H "Authorization: Bearer <access_token>"
```

```json Response
{
  "project_ref": "ca1deb46032e8010",
  "schema_name": "public",
  "db_pooled": {
    "host": "pool.example.com",
    "port": 5432,
    "database": "ca1deb46032e8010",
    "user": "{secret}"
  },
  "db_direct": null
}
```
</CodeGroup>

The response does not include the password or the login username (`{secret}`).
Both are only shown at project creation time (`db_connection_string` in the
create project response) or after a credential rotation.

## PgDog pooler

<Note>
PgDog is an optional component. It is enabled when `[pgdog].public_host` is
set in the MimDB configuration. When PgDog is disabled, `db_pooled` will be
`null` in the connection info response.
</Note>

PgDog is a PostgreSQL connection pooler that sits between your application and
the database. MimDB manages it as a supervised sidecar process (similar to how
it manages PostgREST).

In the default pooling mode (`per_project_role`), each project gets its own
dedicated PostgreSQL database and a login role (the secret half of the
UUIDv8-derived ref) with a unique SCRAM-SHA-256 credential. Clients
authenticate directly to PostgreSQL through PgDog using that credential - no
credential tables or auth queries are needed.

PgDog uses transaction-mode pooling, which means a server connection is only
held for the duration of a transaction, not for the entire client session. This
allows a small number of server connections to serve many concurrent clients.

## Rotate credentials

`POST /v1/platform/projects/{id}/rotate-db-credential`

Generates a new password for the project's PostgreSQL login role and updates
`pg_authid`. The new password is returned once and never stored. All existing
connections using the old password will need to reconnect with the new one.

<CodeGroup>
```bash Request
curl -X POST http://localhost:9000/v1/platform/projects/01970000-0000-7000-8000-000000000002/rotate-db-credential \
  -H "Authorization: Bearer <access_token>"
```

```json Response
{
  "password": "a3f9b2c1d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1"
}
```
</CodeGroup>

<Warning>
Credential rotation is immediate and irreversible. All active connections using
the old password will fail authentication after the rotation. Update your
connection strings before rotating in production, or coordinate a brief
reconnection window with your application.
</Warning>

## Connection string format

Build a standard `postgresql://` URI from the connection info response:

```
postgresql://{secret}:{password}@{host}:{port}/{ref}?sslmode=require
```

- `{secret}` - the login username shown once at project creation
- `{password}` - the credential shown at creation or after a rotation
- `{ref}` - the 16-character project ref, which is also the database name
- Tables live in the `public` schema; no `search_path` configuration is needed

Example with values filled in:

```
postgresql://032e8010ca1deb46:a3f9...@pool.example.com:5432/ca1deb46032e8010?sslmode=require
```

`sslmode=require` is mandatory when connecting through PgDog in production.
PgDog uses passthrough authentication - your password travels from your client
to PgDog in plaintext over the TLS layer before being forwarded to PostgreSQL
as a SCRAM-SHA-256 exchange. Without TLS, the password is exposed in transit.

## Direct vs pooled

| | Pooled (PgDog) | Direct (PostgreSQL) |
|--|----------------|---------------------|
| **Port** | 5432 (session pooler), 6543 (transaction pooler) | 5432 |
| **Connection limit** | Many clients share few server connections | Each client uses one server connection |
| **Transaction mode** | Yes - connection returned after each transaction | No - connection held for session duration |
| **Best for** | Application servers, microservices, short-lived tasks | Migrations, schema management, long-running analytics |
| **Prepared statements** | Extended protocol (default) | Full prepared statement support |
| **Availability** | Only when PgDog is enabled | Only when `DirectHost` is configured |

Use the pooled connection for your application server. Use the direct
connection for database management tools (psql, pg_dump, migration frameworks)
that hold connections longer or require features not supported in transaction-mode
pooling.
