# Projects

A project is the core multi-tenant unit in MimDB. Each project gets its own
isolated PostgreSQL schema, authentication tables, JWT signing key, API keys,
storage signing key, and PostgreSQL roles. Everything project-scoped - REST
API, auth, realtime, storage - operates within that isolation boundary.

## Create a project

`POST /v1/platform/projects`

Creating a project triggers the full provisioning sequence:

1. Generates an 8-character hex `ref` (e.g., `a1b2c3d4`)
2. Creates a PostgreSQL schema named `project_{ref}`
3. Creates three PostgreSQL roles: `project_{ref}_anon`, `project_{ref}_authenticated`, `project_{ref}_service`
4. Applies the tenant template (auth tables, RLS policies)
5. Generates an HS256 JWT signing key stored in `_mimisbrunnr.jwt_signing_keys`
6. Derives the `anon` and `service_role` API keys from that signing key
7. Generates a storage signing key for signed URLs
8. If PgDog is configured in Tier 1 mode, provisions a per-project database credential

The entire sequence runs in a single database transaction. If any step fails,
all changes are rolled back - you will never end up with a partially-provisioned
project.

<CodeGroup>
```bash Request
curl -X POST http://localhost:9000/v1/platform/projects \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{
    "org_id": "01970000-0000-7000-8000-000000000001",
    "name": "My App"
  }'
```

## Get project

`GET /v1/platform/projects/{projectID}`

Retrieves a project by its UUID. Use the `id` from the create response (not
the `ref`).

<CodeGroup>
```bash Request
curl http://localhost:9000/v1/platform/projects/01970000-0000-7000-8000-000000000002 \
  -H "Authorization: Bearer <access_token>"
```

```json Response
{
  "id": "01970000-0000-7000-8000-000000000002",
  "org_id": "01970000-0000-7000-8000-000000000001",
  "ref": "a1b2c3d4",
  "name": "My App",
  "schema_name": "project_a1b2c3d4",
  "region": "",
  "status": "active",
  "settings": {},
  "created_at": "2025-01-01T00:00:00Z",
  "updated_at": "2025-01-01T00:00:00Z"
}
```
</CodeGroup>

## Project ref vs project ID

MimDB uses two distinct identifiers for every project:

| Identifier | Format | Used in |
|------------|--------|---------|
| `id` | UUID v7 (e.g., `01970000-0000-7000-8000-000000000002`) | Platform management endpoints (`/v1/platform/projects/{id}/...`) |
| `ref` | 8-character hex (e.g., `a1b2c3d4`) | Data plane API routes (`/v1/rest/{ref}/`, `/v1/auth/{ref}/`, `/v1/realtime/{ref}`, etc.) |

<Note>
The `ref` is embedded in JWT API keys as the `aud` (audience) claim. This is
how the auth cascade knows which project's signing key to use when validating a
token. The `id` (UUID) is the internal primary key used for database relations.
Keep both - you need the `ref` for day-to-day API calls and the `id` for
platform management operations.
</Note>

## Project status

Projects move through a lifecycle managed by the provisioning system:

| Status | Meaning |
|--------|---------|
| `provisioning` | Schema, roles, and keys are being created. The project is not yet usable. |
| `active` | Fully provisioned. All API endpoints are operational. |
| `paused` | Project is suspended. API requests return errors. (Reserved for future billing/quota enforcement.) |
| `error` | Provisioning failed partway through. Contact support or delete and recreate. |
| `deleted` | Soft-deleted. Schema is retained briefly for recovery before permanent deletion. |

In practice, provisioning is synchronous - a successful `POST /v1/platform/projects`
response means the project is already `active`. The `provisioning` and `error`
states are present in the schema for future async provisioning paths.

<Warning>
Project deletion is permanent and cannot be undone after the retention window
expires. Deleting a project destroys the PostgreSQL schema, all tables and data
within it, all auth users, all uploaded storage files, and all signing keys.
There is no soft-delete recovery API exposed in the current version.
</Warning>
