# Realtime

MimDB Realtime lets clients subscribe to table-level change events over a
WebSocket connection. Every INSERT, UPDATE, and DELETE that passes your
Row-Level Security policies is delivered to subscribed clients in real time,
with no polling required.

## Architecture

Each project has one **Hub** - a long-lived goroutine that owns all WebSocket
clients for that project. A hub is created on the first client connection and
torn down 30 seconds after the last client disconnects (the grace period
prevents churn from page navigation or brief network drops).

The hub receives change events from PostgreSQL via **LISTEN/NOTIFY**. When a
subscribed table is written to, a trigger fires and `NOTIFY`s the hub's
dedicated channel (`{ref}_changes`). The hub re-queries the changed row
under the client's database role to enforce Row-Level Security, then fans the
result out to all matching subscribers.

```
Client A ──┐
           ├── WebSocket ──► Hub ──► PG LISTEN/NOTIFY ──► PostgreSQL
Client B ──┘                  │
                               └── re-query (RLS enforced)
```

Multiple server instances listen on the same PostgreSQL NOTIFY channel, so
events broadcast to all connected clients regardless of which instance they
are connected to.

## Connection limits

Connection limits protect the server from being overwhelmed. The defaults
from `configs/mimisbrunnr.example.toml` are:

| Limit | Default | Config key |
|-------|---------|------------|
| Clients per project | 200 | `max_clients_per_project` |
| Clients globally | 5,000 | `max_clients_global` |
| Subscriptions per client | 100 | `max_subscriptions_per_client` |

<Note>
When the per-project limit is reached, new connection attempts receive a
`503` HTTP response with error code `RT-0300` before the WebSocket upgrade
completes. When the subscription limit per client is reached, the subscribe
message is rejected with error code `RT-0008` over the open connection.
</Note>

You can tune these values in the `[realtime]` section of your configuration:

```toml
[realtime]
max_clients_per_project = 200
max_clients_global      = 5000
```

## Authentication

The auth cascade for realtime mirrors the rest of the data plane: JWT Bearer
token, then API key, then anonymous (`anon` role).

For WebSocket connections, pass the credential as a query parameter because
browser WebSocket APIs do not support custom request headers:

```
ws://localhost:9000/v1/realtime/{ref}?apikey=<jwt_or_api_key>
```

You can also send the `Authorization: Bearer <token>` header on the initial
HTTP upgrade request from environments that support it (server-to-server
connections, custom WebSocket clients).

The resolved role determines which rows are visible via Row-Level Security.
A client authenticated with the `anon` key sees only rows that your RLS
policies permit for the `anon` role.
